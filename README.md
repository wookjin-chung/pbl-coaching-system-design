# PBL Coaching System Design

**PBL 코칭 서비스의 시스템 설계, 내부 기준, 실행 메커니즘 정의 레포지토리**

---

## 이 레포의 정체성 (What this repository is)

이 레포는 **PBL(Project-Based Learning) 코칭 서비스가 어떻게 작동해야 하는지에 대한  
가장 상위의 설계 기준(Source of Truth)** 을 정의합니다.

여기에 포함된 문서들은:

- 코드 구현 이전의 **시스템 헌법**
- UI/UX 이전의 **행동·판단·자각 구조**
- 기능 명세 이전의 **역할 경계와 금지 규칙**

을 다룹니다.

👉 이 레포는 **구현을 설명하지 않습니다.**  
👉 이 레포는 **정답을 제시하지 않습니다.**

이 레포는  
**“판단이 일어날 수 있도록 환경을 어떻게 설계해야 하는가”**를 정의합니다.

---

## 이 레포가 다루는 것 (Scope)

이 레포는 다음을 다룹니다.

### 1️⃣ PBL 코칭의 핵심 철학
- PBL을 “결과 중심 학습”이 아닌  
  **기록·실험·판단·자각의 순환 구조**로 정의
- 인간의 능력이  
  *기억 → 판단 → 자각* 으로 이동하는 흐름에 대한 관점

---

### 2️⃣ AI의 역할과 경계
- AI는 **판단의 주체가 아님**
- AI는 다음을 절대 하지 않음:
  - “이 판단은 옳다 / 틀리다”
  - “이 단계로 가야 한다”
  - “정답은 이것이다”
- AI의 역할은:
  - 기록이 사라지지 않게 정리하고
  - 지금 필요한 시간 축(과거/현재/미래)을 비추며
  - 도움이 될 수 있는 **사고 렌즈를 제안**하는 것

---

### 3️⃣ Path Awareness Layer (경로 인식 장치)
- 탐색–몰입–자각–정리 위에 얹히는 **5번째 레이어**
- 사용자가:
  - 어디서 시작했는지
  - 지금 어디에 있는지
  - 어디로 가고 있는지를
  **스스로 인식하도록 돕는 장치**

---

### 4️⃣ 메인 서비스 실행 메커니즘
- PBL 단계 구조 × 코칭 시나리오(서비스 모드)를  
  **서로 다른 축으로 배열한 실행 구조**
- 코칭을 “정답”이 아니라  
  **상황별 렌즈(모드)** 로 제공하는 방식

---

### 5️⃣ UI 상태 구조와 내부 로직
- 화면은 결과를 보여주는 도구가 아니라  
  **자기 인식을 촉발하는 장치**
- UI 상태, 시간 축 노출, 기록 흐름에 대한 설계 기준

---

### 6️⃣ API / 데이터 / 운영 기준 (설계 관점)
- API 계약(OpenAPI)은 **행동 계약**
- 숫자 대신 **신호(signal)** 를 다루는 데이터 관점
- 프라이버시 / 비식별화 원칙
- MVP 단계에서 **절대 만들지 않을 것(out-of-scope)** 정의

---

## 이 레포가 다루지 않는 것 (Out of Scope)

이 레포는 다음을 **의도적으로 다루지 않습니다.**

- 구체적인 UI 디자인 파일(Figma 시안)
- 프론트엔드/백엔드 구현 코드
- 상용 요금 정책 / 과금 로직
- 개인별 인간 코치 개입 시나리오
- 특정 프레임워크/라이브러리 선택

👉 구현과 운영은 **이 레포를 기준으로 파생**됩니다.

## 변경 원칙 (Change Policy)

- 이 레포의 변경은 **느리고 보수적으로** 이루어집니다.
- 구현 변경은 허용되지만,
  **설계 변경은 반드시 명시적 이유를 동반**해야 합니다.
- 모든 문서는:
  - “이 변경이 판단의 주체를 침해하는가?”
  - “자각을 약화시키는가?”
  를 기준으로 검토됩니다.
