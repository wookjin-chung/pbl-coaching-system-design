## 이 레포는 무엇인가요?

이 레포는 **PBL(Project-Based Learning) 코칭 서비스가  
어떤 원칙 위에서, 어떤 구조로, 어떻게 작동해야 하는지**를 정의하는  
가장 상위의 **설계 기준(Source of Truth)** 입니다.

여기에 담긴 문서들은:

- 구현 코드보다 오래 살아남아야 하는 기준이며
- UI나 기능보다 먼저 합의되어야 할 구조이고
- “무엇을 만들 것인가”보다  
  **“무엇을 판단의 주체로 남겨둘 것인가”**에 대한 선언입니다.

이 레포는 **정답을 제시하지 않습니다.**  
이 레포는 **판단을 대신하지 않습니다.**

대신,  
👉 **판단이 스스로 일어날 수 있도록 환경을 어떻게 설계할 것인가**를 다룹니다.

---

## 왜 이 레포는 공개(Public)되어 있나요?

이 레포는 의도적으로 **Public** 입니다.

이곳에는:
- 구현 코드가 없고
- 상용 알고리즘이 없으며
- 민감한 운영 로직도 없습니다.

대신 누구나 검토할 수 있어야 하는:
- 설계 철학
- 구조적 기준
- AI 역할과 경계

만이 존재합니다.

이 서비스는  
“우연히 만들어진 기능 묶음”이 아니라  
**의식적으로 설계된 시스템**임을 숨기지 않습니다.

---

## 이 설계가 출발하는 핵심 질문들

이 시스템은 다음과 같은 질문들에서 출발합니다.

- 인간의 능력은  
  **기억 → 판단 → 자각**의 방향으로 이동하고 있는가?
- AI는 판단을 대신해야 하는가,  
  아니면 **판단이 일어날 공간을 지켜야 하는가?**
- 학습은 결과의 축적인가,  
  아니면 **자신이 걸어온 경로를 인식하는 과정**인가?
- 사람들은 왜 AI와의 대화를 중단하는가?
- 무엇이 그들을 다시 돌아오게 만드는가?

이 레포의 모든 문서는  
이 질문들에 대한 **구조적 대답**입니다.  
설명이나 수사가 아니라, **설계로서의 대답**입니다.

---

## 이 레포가 정의하는 것 (Scope)

### 1️⃣ PBL 코칭의 철학적 기반
- PBL을 “결과 중심 학습”이 아닌  
  **탐색–몰입–자각–정리**의 순환 구조로 정의
- 학습과 성장이  
  외부 지식 확장에서 **내부 자각 강화**로 이동한다는 관점

---

### 2️⃣ AI의 역할과 경계
- AI는 **판단의 주체가 아님**
- AI는 다음을 절대 하지 않음:
  - “이 판단은 옳다 / 틀리다”
  - “이 단계로 가야 한다”
  - “정답은 이것이다”
- AI의 역할은 오직 다음 세 가지:
  - 기록이 사라지지 않게 정리하고
  - 지금 필요한 **시간 축(과거·현재·미래)** 을 비추며
  - 도움이 될 수 있는 **사고 렌즈를 제안**하는 것

---

### 3️⃣ Path Awareness Layer (경로 인식 장치)
- 탐색–몰입–자각–정리 위에 얹히는 **5번째 레이어**
- 사용자가 스스로:
  - 어디서 시작했는지
  - 지금 어디에 있는지
  - 어디로 가고 있는지를
  **인식하도록 돕는 구조**

이 레이어는  
동기 유발 장치가 아니라  
**자기 인식 장치**입니다.

---

### 4️⃣ 메인 서비스 실행 메커니즘
- PBL 단계 구조 × 코칭 시나리오(서비스 모드)를  
  **서로 다른 축으로 배열한 실행 구조**
- 코칭은 “정답”이 아니라  
  **상황별 렌즈**로 제공됩니다.

---

### 5️⃣ UI 상태 구조와 내부 로직
- UI는 결과를 보여주는 화면이 아니라  
  **자기 인식을 촉발하는 장치**
- 화면은 항상:
  - 지금 사용자가 서 있는 위치
  - 이미 지나온 경로
  - 아직 열려 있는 방향
  중 일부만을 **의도적으로 선택해 노출**합니다.

---

### 6️⃣ API · 데이터 · 운영 기준 (설계 관점)
- API 계약(OpenAPI)은 **기능 계약이 아니라 행동 계약**
- 숫자보다 **신호(signal)** 를 우선하는 데이터 관점
- 프라이버시 및 비식별화 원칙
- MVP 단계에서 **절대 만들지 않을 것(out-of-scope)** 정의

---

## 이 레포가 의도적으로 다루지 않는 것

이 레포는 다음을 **의도적으로 포함하지 않습니다.**

- 구체적인 UI 디자인 시안
- 프론트엔드 / 백엔드 구현 코드
- 요금 정책 / 과금 로직
- 개인별 인간 코치 개입 시나리오
- 특정 프레임워크나 기술 스택 선택

👉 구현과 운영은  
**이 레포의 설계를 기준으로 파생**됩니다.

---

## 이 레포는 어떻게 읽으면 좋을까요?

처음 방문했다면 다음 순서를 권장합니다.

1. `philosophy/`  
   → 이 시스템이 **왜 이런 구조를 갖는지** 이해합니다.
2. `system/`  
   → 서비스가 **어떻게 실행되는지**를 봅니다.
3. `ui/`  
   → 자각이 어떻게 화면으로 드러나는지 확인합니다.
4. `api/`  
   → 시스템 간 행동 계약을 봅니다.
5. `operations/`  
   → 이 설계가 **어디까지를 목표로 하는지** 확인합니다.

## 문서 구조 (Directory Overview)

이 레포는 다음과 같은 구조로 설계 문서를 관리합니다.

### philosophy/
- PBL 코칭의 핵심 철학
- AI 역할과 개입 경계
- 판단·자각·기록 개념 정의

### system/
- 메인 서비스 실행 메커니즘
- Path Awareness Layer
- 코칭 시나리오(서비스 모드) 구조

### ui/
- UI 상태 머신 설계
- 시간 축(과거·현재·미래) 노출 기준
- 경로 인식 UI 구조

### api/
- OpenAPI 계약
- 시스템 간 인터페이스 설계
- 행동 기반 API 원칙

### operations/
- 프라이버시 및 비식별화 원칙
- MVP 범위 및 Out-of-Scope 정의
- 운영 및 변경 관리 기준

## 변경 원칙 (Change Policy)

- 이 레포의 변경은 **느리고 보수적으로** 이루어집니다.
- 구현은 빠르게 바뀔 수 있지만,  
  **설계는 쉽게 바뀌어서는 안 됩니다.**
- 모든 변경은 다음 질문을 통과해야 합니다.
  - 이 변경이 **판단의 주체를 침해하는가?**
  - 이 변경이 **자각을 약화시키는가?**
